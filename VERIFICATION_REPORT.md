# Отчёт проверки (self-check + adversarial workflow)

> Результат строгой перепроверки парсинга и всей логики «со всех сторон» (6 ревьюеров + верификация каждой серьёзной находки).
> 55 находок, 17 high/critical проверены. Кода ещё нет — **все правки делаем на этапе реализации**, это чеклист «как правильно».
> Дата: 2026-07-02.

---

## 1. Общий вердикт

- **Парсинг верный по сути.** Структура (66 вариантов, блоки по 4 столбца, 22×3 темы, строки 3–9 = шаги, 5 общих сообщений + разбор + картинка, 66 картинок-якорей → 30 уникальных файлов) перепроверена по реальным файлам и подтверждена.
- **Один нюанс в моём допущении:** номера карт **смешанные** — «0. Дурак» арабская, а остальные 21 — **римские (I…XXI)**. Мой скрипт извлёк список карт корректно (использовал шаблон, ловящий и римские), но **импортёр обязан сопоставлять карты по НАЗВАНИЮ, а не по номеру** (иначе наивный `\d+` возьмёт только 3 из 66).
- **Архитектура (Вариант A) рабочая и сделает то, что задумано** — но по текущим планам «в лоб» писать нельзя: нужно закрыть 4 класса реальных дефектов (импорт, подстановка имени, планировщик, состояния воронки). Все — превентивные.
- **Итог:** парсинг корректен после правки «карты по имени»; дизайн заработает после реализации фиксов ниже.

## 2. Подтверждённые проблемы (с фиксами, по компонентам)

### CRITICAL — импорт данных

- **C1. Карту определять по каноническому НАЗВАНИЮ, не по номеру.** 0 — арабская, I…XXI — римские. Хардкодим 22 канонических имени (Дурак…Мир), первую строку r8 парсим `^\s*([0-9IVXLC]+)\.\s*(.+?)\.?\s*$` (MULTILINE), название нормализуем (trim, ё→е, убрать точку) и матчим по канону → 66/66. Номер храним только как атрибут отображения.
- **C2. Картинка→вариант — строго по якорю столбца.** В `xl/drawings/drawing1.xml` ровно 66 якорей на строке 7, столбцы {2,6,…,262}; `variant_idx=(col-2)//4`. 66 ссылок ведут к 30 файлам (14 карт — одна картинка на 3 темы, 8 карт — по 2 разных). Импорт: читаем `from-col` → resolve `r:embed`→rels→media; ключ — столбец (не порядок из `ws._images`!). Храним 30 блобов по md5, кэшируем 30 `file_id` на 66 вариантов. Fail-loud проверки: 66 якорей, все row=7, столбцы = {2+4i}, 30 md5.

### HIGH — импорт/контент

- **H1. Имя подставлять ТОЛЬКО в r6 (интро).** Все 3 «ИМЯ» в r8 — это существительное «имя/репутация» (не плейсхолдер!); в r4 — инструкция; в vi46 есть строчное «имя». Никакого глобального `replace('ИМЯ')`. На импорте меняем только первое `^\s*ИМЯ\s*,` в r6 на `{name}`; r4/r8/r9 — байт-в-байт. Тест: r8 всех 66 не изменился.
- **H2. openpyxl грузить в ПОЛНОМ режиме** (`load_workbook(path)`), иначе `_images` пуст и все 66 картинок молча пропадут. Ассерт `len(ws._images)==66`.

### HIGH — движок воронки

- **H3. Обработать ответ во время TRIGGERED** (клиент ответил до отправки приветствия на +7 мин). Сейчас такой ранний ответ игнорится и бот переспрашивает имя — явный «палёж». Фикс: в TRIGGERED тоже ловим ответ (с дебаунсом), пишем name/question; вопрос об имени задаём **только если `name IS NULL`**. Никогда не переспрашивать уже данное имя.

### HIGH — планировщик (общий корень: «отправка + пометка в одной транзакции» физически невозможна)

- **H4. Разделить сетевую отправку и запись в БД; выбрать at-least-once / at-most-once на шаг.** Send нельзя держать внутри SQLite-транзакции. Паттерн **reserve → send → confirm** (outbox). Приветствие/работа/диагностика = at-least-once (можно переслать). **CTA = at-most-once** (пишем sent_log/intent ДО вызова API, при неоднозначной ошибке не повторяем). Убрать из плана ложную гарантию «не дублируется».
- **H5. Следующий шаг цепочки вставлять в ТОЙ ЖЕ транзакции, что помечает текущий отправленным** (+ watchdog). Иначе краш между «working sent» и вставкой «diagnosis» = клиент навсегда завис без шагов (хуже дубля — невидимо). Это чистые записи в БД → один commit возможен. Watchdog: клиент в TRIGGERED/PROCESSING/DIAGNOSING без pending-шага и со старым updated_at → пере-планировать/алерт.
- **H6. Атомарный «захват» шага + `UNIQUE(client_id, step_name)` + статус `sending`.** `UPDATE steps SET status='sending' WHERE id=? AND status='pending'` (rowcount==1 → продолжаем). Дубли делаем невозможными (INSERT..ON CONFLICT DO NOTHING), а не «детектируем после отправки». Один экземпляр процесса (flock; systemd ИЛИ Docker, не оба).
- **H7. Перед необратимой отправкой перечитать состояние клиента** (гонка «отмена vs поллер»). Сначала typing/upload-сон, потом свежее чтение `clients.state`; если HANDOFF/STOPPED/COMPLETED/BLOCKED — отменить. Единый `cancel_pending(client_id)` из handoff/STOP/оператора/обработчика ошибок. В Business «клиент заблокировал» видно только при отправке → 403/PEER_ID_INVALID → STOPPED + cancel.

### MEDIUM-HIGH — планировщик

- **MH1. TTL просроченных шагов + живой guard 24ч + стартовая сверка.** «Просрочка отправляется на следующем тике» — без верхней границы: после долгого простоя приветствие/CTA придёт через часы (жутко), а при простое >24ч от последнего входящего отправка вообще упадёт (`BUSINESS_CHAT_INACTIVE`). Фикс: TTL на шаг (greeting ~15м, working ~20м, diagnosis ~30м, CTA ~5м) — за пределами пропустить/в терминальное состояние; перед каждой отправкой требовать `now - last_incoming_at < 23ч`; `last_incoming_at` обновлять на каждом входящем.

### MEDIUM — планировщик/схема/API (проверенные факты)

- Шаги ключевать **по позиции строки** (r3…r9), не по тексту тайминга (у vi1 в r8 тайминг — пробел; «сразу после» повторяется). Тайминг парсить только для задержек, лениво, с дефолтом.
- **Всё слать plain-text (`parse_mode=None`)**; НЕ ставить глобальный `DefaultBotProperties(parse_mode=...)` — в текстах есть `.`, `-`, `(`, `!`, что сломает MarkdownV2 у 100% сообщений.
- `run_at/sent_at` хранить как **INTEGER epoch UTC**, сравнивать численно.
- В SQLite **нет `SELECT..FOR UPDATE`** — использовать optimistic `version` CAS + атомарный захват статуса.
- **Не держать транзакцию во время сетевой отправки/сна** (WAL — один писатель, иначе `SQLITE_BUSY`). Каждый шаг — отдельная asyncio-задача, чтобы typing не сдвигал 10-сек тик.
- Права проверять на подключении: `connection.rights.can_reply/can_read_messages` (Bot API 9.0), не читать устаревший `can_reply`.
- Явно перечислить `allowed_updates` (`business_connection`, `business_message`, + edited/deleted при необходимости).
- Бэкап WAL — не «копия файла», а online-backup / `VACUUM INTO` / litestream. Возможно `synchronous=FULL` (на этом объёме бесплатно).
- Политика ретраев для `steps.attempts`: RetryAfter → `run_at=now+retry_after`; transient → бэкофф; permanent → отмена; хранить last_error.

### MEDIUM — поведение воронки

- **Fallback без имени:** валидировать извлечённое имя (буквы, длина 2–20, не эмодзи/url); иначе интро без вокатива (убрать «ИМЯ , »), не подставлять мусор.
- **Алгоритм извлечения имени:** срезать приветствия, брать токен до первой запятой / первое слово с заглавной; всю реплику хранить как `question`; при неуверенности — вариант без имени.
- **Ре-триггер вернувшихся:** если состояние COMPLETED/STOPPED/HANDOFF и активность старше 24–48ч — сброс с новым выбором из мешка; идемпотентность неймспейсить `(client_id, run_id, step_name)`.
- **Буфер быстрых нескольких сообщений** (окно 3–5с) перед парсингом ответа.
- **Ответы во время PROCESSING/DIAGNOSING:** добавить переходы «ответ/намерение купить» во все состояния после ответа, не только CTA_SENT (отменять шаги и звать оператора, а не слать канон-CTA поверх живого вопроса).
- **Приоритет кодового слова:** роутер «состояние важнее»: stop/handoff > кодовое слово (только если IDLE) > обработка по состоянию.

### LOW

- Пробелы перед пунктуацией (« ,» « .») — авторский стиль → см. открытые решения.
- Сохранять эмодзи/переносы (UTF-8, plain); порядок +15мин: r6 интро → фото → r8 разбор.
- Гигиена джиттера/чтения: readBusinessMessage с тихими часами + задержкой; «тяжёлые хвосты» задержек; иммедиат-ответ на кодовое слово (10–60с «ща гляну ✋»), чтобы не молчать 7 минут.
- Список фраз-отписок (по всему сообщению, не подстрока) → STOPPED + cancel.
- Поведение при edited/deleted business-сообщениях.

## 3. Подтверждено корректно (оставляем как есть)

- **xlsx — единственный нужный источник** (и тексты, и картинки). docx (170 МБ) **не нужен** для рантайма/импорта (хрупкий парсинг, конфликтующая модель) — только как справка. xlsx JPEG-и **2048², ~0.5 МБ — качества хватает** для sendPhoto.
- **66 якорей → варианты — чистая биекция**; дедуп до 30 файлов — норм, лечится хранением по md5 + кэш file_id.
- **Фото (короткая/пустая подпись) → полный разбор одним `sendMessage`** — правильно и нужно: все 66 разборов > 1024 (лимит подписи), но < 4096 → резать по длине не надо.
- **file_id переиспользуется + fallback на диск** — уже в дизайне, корректно (file_id постоянны; истекает только download-URL из getFile).
- **Окно 24ч (can_reply) на счастливом пути безопасно** (весь цикл ~30 мин). API-факты (business_connection_id на send/sendChatAction, клиент не видит бота, readBusinessMessage, allowed_updates) подтверждены.
- **Входящий-гейт + opt-in внутри инициированного клиентом окна 24ч — «зелёная зона» Telegram**; это + ~30/день + джиттер — и есть настоящая защита от бана.
- **SQLite/WAL + busy_timeout — верный выбор** под эту нагрузку.

## 4. Ложные тревоги / переоценено (снято)

- **«file_id не переиспользуется в business-соединении» → ложная тревога.** Дизайн уже держит диск как истину и перезаливает при ошибке; file_id постоянны. Ничего не усложняем.
- **«Идентичный текст = бан-отпечаток анти-спама» → переоценено**, переформулировано в **medium (правдоподобность персоны / гигиена жалоб)**, не «фильтр-эвейжн». Enforcement Telegram — по жалобам + ручная модерация исходящего спама незнакомцам, не по входящим джиттер-ответам. Пул перефразировок и реализм печати оставляем, «экзистенциальную» рамку убираем.
- **«docx 61/66 из-за римских цифр» → опровергнуто** (это склейка runs в python-docx). Суть C1 (не джойнить по номеру) остаётся.
- **«CTA рассылает ссылку всем» → неверно для нашего дизайна:** в r9 ссылки нет, цену/ссылку даёт живой оператор после намерения.

## 5. Открытые решения (нужен ваш выбор; в скобках — рекомендация)

1. **Доставка разбора:** одним сообщением (проще) vs 2–4 «пузыря» (человечнее, нужно для реалистичной печати). *(Рекомендую пузыри.)*
2. **Ранний ответ (TRIGGERED):** A — оставить +7-мин приветствие, но без переспроса имени, с персональным «ага, вижу» *(рекомендую, сохраняет «занятой» темп)*; B — отменить приветствие, ответить сразу.
3. **Нормализация текста:** слать как есть (авторский стиль с « ,» « .») vs аккуратная нормализация пробелов на импорте с ручной сверкой. *(Рекомендую нормализацию — для «человечности».)*
4. **Иммедиат-ответ на кодовое слово** (10–60с) vs полные 7 минут тишины до первого контакта.
5. **Ре-триггер вернувшихся:** cooldown + run_id *(рекоменд.)* vs одна воронка на клиента навсегда.
6. **Пул перефразировок общих сообщений** (3–5 на шаг) *(рекоменд. для персоны)*; смягчать ли CTA «Решай 😈».
7. **synchronous NORMAL vs FULL** *(FULL, если строгий at-most-once CTA должен переживать отключение питания).*
8. **8 «расходящихся» карт** (где картинка разная по темам) — ручная QA, что это намеренное разное арт, а не пересжатие.
9. **Day-2 фоллоу-ап** в будущем — только при свежем входящем в пределах 24ч (никогда «холодно»). *(Рекоменд.: пока нет.)*
